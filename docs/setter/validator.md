# Validator

## Main Goals

The main goals of this stage are:
- Write a validator that accepts exactly the inputs allowed by the statement.
- Make the validator easy to read enough for testers to review.
- Write validator tests that target the fragile parts of the validator.

::: info Polygon checklist for this stage
- Upload the validator in Polygon.
- Write validator tests, and check that the validator passes all validator tests.
- **Commit the changes in Polygon.**
:::

## Writing the Validator

::: danger Validator mistakes are critical
If the validator accepts invalid tests, official tests may fail to satisfy the statement. This is one of the most serious failures that can happen in a contest.
:::

The validator is a program that checks whether input files follow the specifications written in the statement. Since test data is often generated by scripts, a correct validator is essential for preventing invalid official tests.

For validator implementation, the following two principles are **extremely important**:

- **Correctness**: check every input constraint written in the statement, exactly as written.
- **Readability**: write the validator so that testers can easily read and verify that it matches the statement.

This manual provides a validator template in [Code Template](../reference/code-template.md). The template is designed to make validators consistent and easy to read. Every problem should use this validator template unless a problem director approves an exception.

For technical details about validator implementation and Polygon usage, see [Polygon Manual: Validator](../polygon/).

::: warning Subtask constraints must also be checked
If the problem has subtasks, the validator should also check whether each test satisfies the constraints of its assigned subtask. See [Code Template](../reference/code-template.md) for the recommended style.
:::

## Writing Validator Tests

Validator tests are an additional safety measure for checking that the validator behaves as intended.

::: tip Focus on complex logic
Validator tests are most useful for constraints that require nontrivial logic, such as checking whether a graph is a tree, whether edges are unique, or whether an array is a permutation.
:::

You may add tests for any input constraint, including the subtask constraints. However, simple range constraints are often better checked by reading the validator, assuming that the official code template is used correctly.

Do not make validator tests unnecessarily large. For example, if the constraint is $n \le 10^5$ and the input contains an array of length $n$, avoid creating a huge validator test only to check the upper bound of $n$. Such a test may be hundreds of kilobytes while adding little value.

## Statement Consistency

At this point, the validator should match the current statement exactly. From now on, keep the statement and validator synchronized:

- If you modify the statement, update the validator if needed.
- If you modify the validator, check whether the statement should also be updated.

::: danger Statement-validator mismatch is a common source of contest errors
A common validator mistake is forgetting to update the validator after editing the statement. Always recheck the statement and validator together after either one changes.
:::

## Before Moving On

::: tip Ready to move on when
- The validator checks all constraints in the statement, exactly as written.
- The validator also accounts for subtasks, if the problem has subtasks.
- The validator is easy for another committee member to read and review.
- The validator passes all validator tests.
- **The changes are committed on Polygon.**
:::
