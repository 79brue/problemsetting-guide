## Caution

::: warning Do not add samples to Polygon yet
Polygon generates sample outputs by running the sample inputs on the main correct solution. Therefore, if you add the sample tests to Polygon, you will not be able to see the rendered statement (including samples) before you add both the checker and MCS.

In this stage, design the samples first. After writing the checker and main correct solution, return to this page and add the samples and sample explanations to Polygon.
:::

## Choosing Samples

Sample tests are part of the statement. They should help contestants:

- check whether they understood the problem correctly;
- understand the input and output format;
- debug their code on small examples.

It is strongly recommended to make samples **friendly**. Friendly samples cover important cases and are useful for debugging, while still being easy to understand.

::: details How to make friendly samples
Some useful ways to make samples friendly are:

1. If the output has explicit cases, such as `YES`/`NO` or `-1` for impossible cases, include samples that cover each important case.
2. Include corner cases that contestants are likely to miss.
3. Keep most samples small enough to understand and debug by hand.
4. Avoid samples that are so small that they only show trivial behavior.
5. If useful, add one moderately large sample, usually around $n \sim 20$, to help contestants catch implementation mistakes.
:::

For most problems, each sample has a single intended answer. Before writing the main correct solution, try to compute the answer for each sample manually if the sample is small enough. You can later use these samples to test the main correct solution.

## Adding Samples

After the checker and main correct solution are ready, add the samples to the tests and mark them as examples. See [Polygon Manual: Tests](../polygon/) for technical details.

After adding the samples, return to the statement page and render the statement in HTML or PDF. Check that the sample inputs are displayed correctly, and the generated sample outputs are correct.

::: details I want to change the sample output
Sometimes, if the problem allows multiple valid outputs or has a special format (e.g. interactive), you may want to change the sample output displayed in the statement.

Common reasons include:

- the output generated by the main correct solution gives too much hint about the intended solution;
- another valid output is easier to understand;
- for an interactive problem, adjusting line breaks shows the interaction protocol better.

:::

## Sample Explanations

For most problems, it is recommended to add sample explanations. A sample explanation may describe:

- what situation the sample input represents;
- how the sample output was computed, or why the sample output is valid;
- for interactive problems, the order in which the interaction happened.

For now, write sample explanations in the `Notes` section. During export, these explanations will be moved below the corresponding samples.

## Before Moving On

- Samples help the reader understand the statement or debug wrong code.
- Useful sample explanations are written in the `Notes` section.
- **The changes are committed on Polygon.**
